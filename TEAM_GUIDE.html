System Overview
What this app does and why it matters
Core idea: Airport baggage systems can't go down — even when the cloud connection fails. This app demonstrates edge-first resilience: a local Couchbase node keeps running independently during an outage, and automatically syncs back to the cloud the moment connectivity is restored.
Architecture
Edge Node (local)localhost:8091
Main / Cloud Nodelocalhost:8092
Sync Gatewaylocalhost:4984
Backend APIlocalhost:8000
Frontendlocalhost:3000
Replication

All bag writes go to the Edge node first. XDCR (Cross Datacenter Replication) continuously pushes changes to the Main/cloud node in the background.

When the connection is cut (simulated outage), the Edge keeps accepting writes locally. When restored, everything syncs automatically — zero data loss.

Key Demo Flows
The 3 most important scenarios to show a client
Flow 1 · Seed & Explore (2 min)

Get the system to a clean demo state and walk through the UI.

1
Open Simulation Controls (◀ left tab) → click SEED TEST DATA
2
Watch event log: "SEEDED 5 flights + 20 bags on Edge [SYNCING to Main]"
3
Show Split-Brain Panel: Edge = Main = 20 bags, status IN SYNC
4
Tour LIVE OPS → Belt Heatmap → Baggage Hall → Workstations
Flow 2 · Outage & Recovery (3 min) ← Most impressive

Demonstrate that the system keeps working with zero downtime during a cloud outage.

1
Simulation Controls → toggle CLOUD SYNC to OFFLINE
2
Add a bag or generate load — notice PENDING sync label on new bags
3
Split-Brain Panel shows Edge > Main — diverged, turning red
4
Point out: app is fully operational — operators can still check in bags, view flights, update statuses
5
Toggle CLOUD SYNC back to ONLINE
6
Watch Split-Brain Panel: Edge = Main, SYNCED — all offline bags replicated with zero data loss
Flow 3 · Passenger No-Show (2 min)

Demonstrate IATA compliance automation.

1
Go to BAGGAGE HALL tab → Passenger Board
2
Pick any passenger with status CHECKED IN
3
Click NO SHOW → confirm
4
Toast appears: "N bags put on hold for offload"
5
Open AlertRail (▶ right tab) → HIGH severity: PASSENGER NO-SHOW alert
6
Bag Table shows affected bags with status ON HOLD

API Quick Reference
Backend endpoints at localhost:8000
MethodEndpointWhat it does
GET/healthSystem health, Edge/Main doc counts, sync status
GET/bagsAll bags
POST/bagsCreate a new bag
PUT/bags/{id}/statusUpdate bag status
GET/flightsAll flights
POST/simulate/seedReset + seed 5 flights, 20 bags, 20 passengers
POST/simulate/offlinePause XDCR replication (simulate outage)
POST/simulate/onlineResume XDCR replication (restore)
POST/simulate/load?count=NBulk generate N bags
GET/passengersAll passengers (optional ?flight_id=SK101)
PUT/passengers/{id}/boardMark passenger as boarded
POST/passengers/{id}/no-showDeclare no-show → auto hold all bags
GET/auditAudit log entries
GET/conflictsUnresolved split-brain conflicts
GET/integrations/liveConveyor belt health, DCS, ACRIS, RFID feed
