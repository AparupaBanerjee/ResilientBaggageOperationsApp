01 · Business Overview

What Problem Does This Solve?

⚠️

The Challenge

Airport terminal systems depend on a central cloud server for flight data, passenger records, and baggage routing rules. When that connection fails — due to a network outage, maintenance window, or cyber incident — staff lose visibility, bags get misrouted, and passengers miss flights.

✅

Our Solution

We deploy a local Couchbase database at every terminal (Edge Node) that keeps operating independently. All data syncs to the central cloud when connected, and automatically reconciles when reconnected — with a human-in-the-loop review when conflicts arise.

🛄

Zero Downtime

Terminal keeps routing registered baggage even with no cloud connectivity

👮

IATA Compliance

Passenger no-show triggers instant bag hold — mandatory aviation safety rule enforced automatically

🧑‍✈️

Human Control

Duty Manager reviews every data conflict — no automated override for safety-critical decisions

02 · System Architecture

How the System Is Structured

🏗️ Control Tower (Cloud)

Central Couchbase Main cluster. Manages all flight schedules, cancellation directives, and receives real-time bag data from all terminals via XDCR replication.

🖥️ Terminal Nodes (Edge)

Each airport terminal runs its own Couchbase instance. Operates fully autonomously — check-in, routing, scanning — even with zero cloud connectivity.

🔄 XDCR Data Link

Cross Data Centre Replication keeps all nodes in sync in real time. When a terminal goes offline, it queues writes locally and syncs when the link is restored.

04 · Demo Scenario 1

Normal Outage & Automatic Recovery

This scenario demonstrates the core resilience capability — baggage operations continue uninterrupted even when the Inter-Terminal Data Link (XDCR) is severed.

1

🔴 Data Link Severed

The XDCR replication between Terminal B and the Control Tower is paused — simulating a network outage, fibre cut, or maintenance window.

Terminal B is now operating in Isolated Mode. Staff see no change — operations continue normally.
2

✈ Bags Keep Moving — Offline

Passengers check in. Bags are scanned, assigned to Loading Bay B2, and registered in the local Couchbase database. All data is queued for sync with SYNC PENDING status.

The system writes locally first. No data is lost. Edge has +3 bags that Main hasn't seen yet.
3

🟢 Link Restored — Auto Sync

The XDCR data link is re-established. All queued bag records are automatically pushed to the Control Tower cloud. Both systems reconcile to a consistent state.

Control Tower now shows Edge = Main = same bag count. IN SYNC ✓ — No human action required.
05 · Demo Scenario 2

Flight Cancellation Conflict — Split-Brain Resolution

This is the core hackathon scenario. The Control Tower cancels a flight while a terminal is offline. When sync restores, the two systems have conflicting states — a human Duty Manager must intervene.

1

🔴 Terminal B Goes Offline (Data Link Severed)

Terminal B loses connectivity to the Control Tower. Bags for Flight SK202 (Paris CDG) continue to be checked in and routed to Loading Bay A2 — staff at Terminal B have no idea anything is wrong.

2

📢 Control Tower Issues Cancellation Directive

A Control Tower operator cancels Flight SK202 — perhaps due to crew unavailability or technical fault. This directive is logged in the Control Tower cloud database. Terminal B never receives it.

⚠ Diverged State: Control Tower says SK202 = CANCELLED. Terminal B still says SK202 = SCHEDULED and is routing bags to the aircraft.
3

🔁 Data Link Restored — Conflict Detected

XDCR replication resumes. The system detects that Terminal B has active bags routing for a flight the Control Tower has cancelled. A conflict alert is raised in the Baggage Operations Dashboard.

Couchbase XDCR uses timestamp-based conflict detection (CAS — Compare-And-Swap). The system flags this as a Operational Discrepancy requiring human review.
4

🧑‍✈️ Duty Manager Reviews & Resolves

The Duty Manager at Terminal B opens the Split-Brain Resolution panel. They can see exactly which bags are affected and choose one of two resolution options.

The system never auto-resolves safety-critical conflicts. A human makes the final call — every decision is audit-logged with operator ID and timestamp.

Resolution Options Available to Duty Manager

☁ Option A: Accept Cloud Directive

Follow the Control Tower cancellation order — IATA-compliant, safety-first approach.

  • Flight SK202 status → CANCELLED
  • All active registered baggage → ON HOLD
  • Bags removed from Loading Bay A2 queue
  • Audit log entry created with Duty Manager ID
  • Passengers notified via DCS system

✎ Option B: Manual Operational Update

Operator override — flight cancelled but bags diverted to a special handling belt.

  • Flight SK202 status → CANCELLED
  • All active registered baggage → Belt D12 (Emergency Divert)
  • Bags physically redirected to temporary staging area
  • Audit log entry created with target belt ID
  • Enables ground crew to manually process bags
06 · Advanced Scenario

Multi-Terminal Network — Control Tower Sends Directive

What happens when the airport has multiple terminals, and only one goes offline when the Control Tower issues a cancellation? This is the realistic large-airport scenario.

✓ Terminal A — No Issue

Received the cancellation directive in real time. Automatically stopped accepting bags for SK202. No human action needed.

⚠ Terminal B — Conflict

Was offline when directive was issued. Kept routing bags for a cancelled flight. Duty Manager must review when link restores.

✓ Terminal C — No Issue

Received the cancellation directive in real time. Halted aircraft loading at the gate. No human action needed.

07 · Demo Scenario 3

Passenger No-Show — IATA Bag Offload Rule

IATA mandates that if a passenger does not board, their checked bags must be removed from the aircraft immediately. This system enforces that rule automatically and in real time.

1

Passenger Checks In

Erik Johansson checks in for SK101 to London Heathrow. His bag is tagged, weighed, and routed to Loading Bay A1. A passenger record is created: CHECKED IN

2

Gate Closes — Passenger Absent

Boarding gate closes. Erik Johansson has not presented his boarding pass. Gate agent marks him as NO SHOW in the system.

3

Automatic Bag Hold (IATA Compliance)

System instantly locates all bags belonging to Erik Johansson on SK101 and sets them to ON HOLD with reason: Passenger No-Show. Belt assignment cleared.

Alert raised in Baggage Operations Centre: "PASSENGER NO-SHOW — BAG HELD FOR OFFLOAD"
4

Ground Staff Retrieve the Bag

Loading crew receives the alert. The bag is physically removed from the aircraft hold before departure. Full audit trail recorded.

Flight SK101 departs on time. IATA security compliance maintained. ✓
