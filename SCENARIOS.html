Quick Reference — What each scenario does (plain English)
01 SEEDFill the airport system with 5 flights and 20 bags so you have data to work with.
02 ADD BAGCheck in one bag for a passenger on a specific flight, just like a real counter agent would.
03 GENERATE LOADAdd many bags at once (up to 50) to simulate a busy check-in period.
04 OUTAGECut the link between the airport terminal and the cloud — bags still get processed locally.
05 RESTOREBring the link back — all bags written during the outage automatically sync to the cloud.
06 MISROUTESend one bag to the wrong conveyor belt on purpose — see the system catch and alert on the mistake.
07 FLAKY MODEMake 30% of bag writes randomly fail — like a database having hiccups — and watch the system handle it.
08 BLACKOUTTurn off all RFID bag scanners for 30 seconds — no tracking data comes in during that window.
09 CORRUPT DATAInsert a broken bag record with missing and invalid fields — see the system flag it without crashing.
10 WRITE STORMFlood the system with 50 bags at the same time to prove it handles peak load without breaking.
11 FLIGHT ACTIONSDelay, cancel, or mark a flight arrived — and watch the system automatically reroute or hold its bags.
12 SPLIT-BRAINCloud cancels a flight while the terminal is offline — when reconnected a human must decide what to do with the bags.
13 NO-SHOWMark a passenger as not boarded — the system instantly puts their bags on hold as required by IATA security rules.

Baseline Operations

01
Seed Initial Data
Populate the Edge database with 5 realistic SAS flights and 20 passenger bags with matching passenger records. This is the starting point for every demo.
BASELINE XDCR SYNC
Steps
1
Open Simulation Controls
Click the slider icon on the left edge of the screen to open the control drawer.
2
Click SEED DATA
The large blue button at the top of the drawer.
POST /simulate/seed
3
Watch the event log
Log shows: SEEDED 5 flights + 20 bags on Edge [SYNCING to Main]
✓ SEEDED message appears in cyan
4
Verify CouchbaseStats panel
Top panel shows: Edge=30, Main=30, status IN SYNC (green).
✓ Both nodes show equal document count
5
Check Baggage Hall tab
20 bags visible across flights SK101–SK505 with statuses check_in / in_transit / loaded.
Expected Outcome
Edge DB: 30 docs (5 flights + 20 bags + 5 routing rules). XDCR replicates all to Main within ~2s. BaggageTable populates with bags across all 5 flights.
Edge writes → XDCR → Main replication
02
Add Bag Manually
Register a single passenger bag for a specific flight. Demonstrates real-time check-in capability and live document creation with XDCR propagation.
OPERATOR ACTION REAL-TIME SYNC
Steps
1

2
Enter passenger name

Bag created on Edge with status=check_in, assigned to correct belt via routing rules. Replicates to Main in ~2s. If offline, sync_pending=true until RESTORE.
API → routing lookup → Edge write → XDCR

Sync & Outage Simulation

04
Simulate Network Outage
Pause XDCR replication between Edge and Main — simulating a WAN link failure between the airport terminal and cloud data centre. Edge continues operating independently.
OUTAGE XDCR PAUSE
Steps
1
Click CLOUD SYNC toggle (ON → OFF)
POST /simulate/offline
2
Observe CouchbaseStats panel
Sync toggle turns RED. Status changes to OUT OF SYNC.
✓ "CLOUD SYNC OFF" label in red
3
Add a bag (optional)
Add a bag while offline. It shows sync_pending=true.
⚠ New bags show PENDING — not yet on Main
4
Verify divergence
Edge count rises; Main count stays the same. Document counts diverge.
✓ Edge=31, Main=30 — OUT OF SYNC
Expected Outcome
XDCR replication paused via Couchbase REST API. Edge writes continue at full speed. Main stays frozen at last-synced state. CouchbaseStats shows OUT OF SYNC in amber/red.
XDCR paused — Edge writes, Main frozen
05
Restore Connection
Resume XDCR replication. All bags written during the outage synchronise to Main automatically. Demonstrates self-healing resilience.
RESILIENCE AUTO-SYNC
Steps
1
Prerequisite: Run Scenario 04 first
System should be offline with diverged doc counts.
2
Click CLOUD SYNC toggle (OFF → ON)
POST /simulate/online
3
Watch CouchbaseStats
Status briefly shows SYNCING (blue), then IN SYNC (green) once doc counts equalise.
✓ SYNCING → IN SYNC within 5–10 seconds
4
Verify sync_pending flags cleared
Bags that were offline now show sync_pending=false in BaggageTable.
Expected Outcome
XDCR resumed. All pending docs replicated to Main. Counts equalise: Edge=Main. No data lost — this is Scenario 04 Case 1 (clean recovery, no conflict).
XDCR resumes → counts equalise → IN SYNC

Flight Operations

11
Flight Actions — Delay, Cancel, Arrive
Operators can update flight status and trigger automatic cascade effects on registered baggage — rerouting, on-hold, and offload to Reclaim (RCL) based on airline policy rules.
FLIGHT OPS CASCADE RULES CANCEL
Steps
1
Open Flight Actions in controls drawer
Select a flight from the dropdown (e.g. SK202 — Dubai).
2
Option A — Delay Flight
Click DELAY FLT → confirm. Status → delayed. Bags flagged.
PUT /flights/SK202/status {status: "delayed"}
✓ FlightBoard shows DELAYED in amber
3
Option B — Cancel Flight
Click CANCEL FLT → confirm. Policy engine runs: bags rerouted to alternate, some on hold, others to RCL.
POST /flights/SK202/cancel
⚠ Irreversible — all bags cascade
4
Option C — Mark Arrived (inbound only)
Inbound flights get ARRIVED button. Sets status=arrived.
5
Read cascade result in log
Shows: "SK202 CANCELLED — 3 rerouted→SK303 · 1 on hold"
Expected Outcome
Cancel triggers automatic bag cascade: bags go to alternate flight if available, else on_hold or offloaded. FlightBoard updates. BaggageTable shows new bag statuses. All changes replicated via XDCR.
Cancel triggers policy cascade on all bags

Split-Brain Conflict Resolution

12
Split-Brain Conflict — Cloud Cancel vs Edge
The flagship demo. Cloud Ops Centre cancels a flight while the terminal is offline (XDCR down). When connection restores, a conflict is detected — a human operator must resolve it manually using one of two strategies.
SPLIT-BRAIN HUMAN RESOLUTION CONFLICT DETECTION
Step-by-Step Flow
1
Seed data first
POST /simulate/seed — ensure flights exist.
2
Take XDCR offline
Click CLOUD SYNC OFF. Terminal is now isolated.
POST /simulate/offline
3
Issue Cloud Cancel
Select a flight (e.g. SK303). Click CLOUD CANCEL.
POST /conflicts/cloud-cancel {flight_id: "SK303"}
⚠ Directive stored — XDCR offline so Edge not updated yet
4
Restore connection
Click CLOUD SYNC ON. XDCR resumes — system now detects conflict.
POST /simulate/online
5
Conflict appears in Split-Brain panel
CONFLICTS tab shows a directive for SK303. Displays: Edge status vs Cloud intent (CANCELLED).
✓ Conflict card appears with directive ID
6A
Resolution: ACCEPT CLOUD
Click ACCEPT CLOUD card → confirms. Flight status → CANCELLED. All bags → ON HOLD for offload.
POST /conflicts/resolve {action: "accept_cloud"}
✓ Flight cancelled, bags on hold
6B
Resolution: MANUAL UPDATE
Enter belt (e.g. D12). Click MANUAL UPDATE. Flight → CANCELLED. Bags → diverted to belt D12.
POST /conflicts/resolve {action: "manual_update", divert_belt: "D12"}
✓ Bags rerouted to D12
Expected Outcome
Human-in-the-loop conflict resolution. Edge is the source of truth; cloud directive is advisory. Operator chooses strategy. All decisions recorded in conflict history with operator ID and timestamp.
3-phase split-brain resolution flow

IATA Compliance — Passenger No-Show

13
Passenger No-Show (IATA Rule)
IATA security regulation requires that if a passenger does not board, their registered baggage must be immediately placed on hold and offloaded before departure. This scenario demonstrates automated compliance enforcement.
IATA SECURITY COMPLIANCE PASSENGER OPS
Steps
1
Go to BAGGAGE HALL tab → Passenger Board
Passenger Board is above the BaggageTable. Select a flight tab (e.g. SK101).
2
Find a passenger with CHECKED IN status
Grey CHECKED IN badge — passenger has not boarded yet.
3
Mark passenger as NO SHOW
Click the NO SHOW button → confirm in the modal. System immediately acts.
POST /passengers/{id}/no-show
⚠ Triggers immediate bag hold — IATA compliance
4
Read the toast notification
"N bags put on hold for offload"
✓ Bags status changes to on_hold instantly
5
Check AlertRail
HIGH severity alert: "PASSENGER NO-SHOW — [Name] — bag [ID] on hold for offload"
6
Optional: Board a passenger first
Click BOARD → status turns green BOARDED. Boarded passengers cannot be no-showed.
PUT /passengers/{id}/board
Expected Outcome
No-show triggers N1QL UPDATE on all active bags for that passenger+flight: status→on_hold, hold_reason=passenger_no_show. AlertRail surfaces HIGH alert. Audit entry recorded. IATA compliance enforced automatically.
No-show → immediate bag hold → alert → audit
